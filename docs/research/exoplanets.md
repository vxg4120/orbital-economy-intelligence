# Exoplanets — Does the Satellite Identity-Graph Engine Transfer to "Where Should We Look for Life"?

**Subject:** whether the orbital-economy engine — *provenance-tracked entity resolution across disagreeing catalogs* + *massive time-series analytics on cheap infrastructure* + *metrics that ride on clean identities* (see `docs/research/landscape.md` and `docs/SPEC.md` §1–2: a 69,878-object cross-catalog identity graph with SCD2 temporal ownership, per-attribute `source_assertion`, an auditable `merge_log`, a ~9.67M-element OMM fact layer, and a conflict/data-quality report) — rhymes with exoplanet science, specifically the question *"where should we focus to find life."*
**Research date:** 2026-07-14. Every claim carries a URL + date; unverifiable claims are tagged `[unverified]`.

**One-line finding:** The three-pillar engine rhymes **partially and unevenly**. Pillar 1 (provenance-tracked reconciliation) rhymes *strongly but has a live incumbent* (Exo-MerCat). Pillar 2 (cheap queryable time-series) rhymes on the data but is **the most-occupied lane** — a serious, funded, cloud-native stack (LSDB/HATS) already lives there. Pillar 3 (metrics on clean identities) is where the **genuine, apparently-unoccupied whitespace** is: **provenance-tracked, disagreement-propagated re-ranking of JWST/HWO habitability + observability targets — "does the target rank flip depending on which catalog you trust?"** That fuses all three pillars onto a genuinely contested, high-stakes decision (which ~25 stars a multi-billion-dollar telescope stares at).

## 1. The catalogs and their disagreements — the entity-resolution rhyme

**The rhyme is real: exoplanet science has NASA-vs-EU-vs-OEC catalog disagreement and a host-star cross-ID tangle that is structurally identical to SATCAT/GCAT/UCS.** But unlike satellites — where I found *no* open provenance-tracked reconciler — exoplanets already have one (Exo-MerCat). That changes the positioning, not the opportunity.

**NASA Exoplanet Archive** — the authoritative catalog, run by NExScI at IPAC/Caltech. **6,319 confirmed planets as of 07/09/2026** (homepage banner + counts page, https://exoplanetarchive.ipac.caltech.edu/ , https://exoplanetarchive.ipac.caltech.edu/docs/counts_detail.html , 2026-07-14). Method split: transit 4,662, RV 1,195, microlensing 281, imaging 98, TTV 40, astrometry 6; mission yields Kepler 2,784, TESS 905, K2 549. Only **2,130 planets have a measured mass vs 4,714 with a radius** — a sparsity that forces the archive to *impute*. Separately it tracks **8,064 TESS Project Candidates (TOIs)** as of 07/06/2026, excluded from the confirmed count. Confirmation criterion: mass/minimum-mass **≤ 30 Jupiter masses**, peer-reviewed, false-positive unlikely (statistical validation counts) `[≤30 M_Jup and wording from secondary summaries — Exo-MerCat/Wikipedia — consistent with but not directly quoted from the archive criteria page; treat as strong-not-verbatim]`.
- **ps vs pscomppars** (https://exoplanetarchive.ipac.caltech.edu/docs/pscp_about.html , 2026-07-14): the **Planetary Systems (PS)** table stores **one row per planet per publication** — each row internally self-consistent but sparse; the **Composite (PSCompPars)** table stores **one row per planet**, composited across papers, "more complete but not necessarily self-consistent." Composite fill order: default solution → most-precise value (ties → most-recent, literature over catalogs) → calculated via mass–radius relations. **The PS table literally *is* a per-attribute conflict corpus** — this is the exoplanet analog of exposing "SATCAT says X, GCAT says Y."
- **Access/license:** IVOA **TAP + ADQL** (CSV/JSON/VOTable), a legacy REST API, bulk pull via TOPCAT/pyVO (https://exoplanetarchive.ipac.caltech.edu/docs/TAP/usingTAP.html , 2026-07-14). **US Government Work / public domain**, cite the dataset DOI. *Frictionless — cleaner than Space-Track's redistribution restriction.*

**exoplanet.eu (Extrasolar Planets Encyclopaedia):** **~8,260 listed objects as of 2026-07-14** (https://exoplanet.eu/home/ , 2026-07-14), spanning **Confirmed / Candidate / Controversial / Retracted**. Mass limit **≤ 60 M_Jup + 1σ** — deliberately more inclusive than NASA's 30 (https://exoplanet.eu/readme/ , 2026-07-14). **License: CC BY 4.0.** The higher mass cutoff + candidate inclusion is the main reason it runs larger than NASA.

**Open Exoplanet Catalogue (OEC):** **still actively maintained** — the GitHub repo does automated **nightly commits (latest 2026-07-14T07:01Z)**, ~5,768 commits (https://github.com/OpenExoplanetCatalogue/open_exoplanet_catalogue , GitHub API, 2026-07-14). Format: **human-readable XML, one file per system**. **License: MIT** (permissive/open). Live planet count **~5,400** `[unverified — inferred; Exo-MerCat logged 5,402 OEC entries Oct 2024]`.

**Why the counts differ** (the same four levers as satellites): (a) **mass cutoff** — NASA ≤30, EU ≤60+1σ, old IAU line 13 M_Jup (the unresolved planet/brown-dwarf boundary); (b) **candidate handling** — EU/TOI carry huge candidate pools, NASA/OEC do not; (c) **validation philosophy** — statistical validation vs RV/transit confirmation; (d) **update-diligence drift** — values copied from papers that get "contradicted…or copied incorrectly." **Quantified spread:** Exo-MerCat v2.0.0 (https://arxiv.org/abs/2502.08473 , 2026-07-14) concatenated **30,034 entries as of 2024-10-25 — EU 9,804, TOI 7,241, NASA 5,785, OEC 5,402, EPIC 1,800.** Confirmed-planet spread today is **NASA 6,319 vs EU ~8,260 — ~30%**, widening once candidates fold in.

**Host-star cross-ID tangle — a known hard problem.** One host carries **HD, HIP, TIC, KIC, KOI, Gaia DR3, and 2MASS** designations, and "the ID in one data release does not always correspond to the same ID in other releases." Reconciliation tooling already exists: **SIMBAD/CDS** (main-ID + alias resolution), **CDS X-Match**, **TOPCAT** (positional match ±1″), **astroquery/pyVO**, and **ExoFOP** TIC↔Gaia DR3 tables. **Exo-MerCat** (https://arxiv.org/abs/2002.01834 , 2026-07-14) automates it: resolve every host to a SIMBAD main ID via aliases, then merge.

**Is the parameter disagreement real and quantifiable? Yes — and it is the strongest part of the rhyme.** Two named "words of caution" exist: RNAAS *"A Word of Caution About Exoplanet Archival Data"* (https://iopscience.iop.org/article/10.3847/2515-5172/abcf4d , 2026-07-14) and MDPI *"A Quantitative Comparison of Exoplanet Catalogs"* (https://www.mdpi.com/2076-3263/8/9/325 , 2026-07-14); an MNRAS study even finds **systematic differences between transit-spectroscopy pipelines "at the catalogue level"** (https://academic.oup.com/mnras/article/531/1/35/7656976 , 2026-07-14). The dominant systematic is **Gaia parallaxes revising stellar radii → planet radii**:
- **Berger et al. 2018** (https://arxiv.org/abs/1805.00231 , 2026-07-14): re-derived **177,911 Kepler stars + ~4,000 planets/candidates**; median stellar-radius precision **≈8%, a 4–5× improvement**, repopulating the radius gap.
- **California-Kepler Survey VII** (Fulton & Petigura 2018, https://arxiv.org/abs/1805.01453 , 2026-07-14): Gaia DR2 + spectroscopy cut stellar-radius error **~11% → ~3%** and planet radii to **~5%**, sharpening the bimodal **radius gap at ~1.3 and ~2.4 R⊕**. Pre-Gaia photometry-only radii were **~40% uncertain**; the mean planet radius shifted **~3.76%** with new Gaia data.

**Net:** for the *same* planet, radius/mass/insolation genuinely disagree by tens of percent across eras and archives, and the disagreement changes scientific classification (rocky vs sub-Neptune; in-HZ vs not). This is a real reconciliation problem — but **Exo-MerCat already occupies the "merge them" lane.** Its method — pick the **lowest-relative-error** value per parameter — *flattens* the disagreement to a single answer, exactly the way GCAT flattens SATCAT/GCAT in satellites. **The un-occupied move is not "merge," it is "expose and propagate the disagreement downstream."** (See §4.)

## 2. The massive-time-series reality — the cheap-engine rhyme

**The data is genuinely extreme, but this is the *most* occupied of the three lanes.** A funded, sophisticated, cloud-native analytics stack (LSDB/HATS, Parquet) is being built for exactly "cheap queryable photometry at scale" right now.

**Volumes (all 2026-07-14):**
- **Kepler/K2 (legacy):** ~200,000 stars, cadences 29.4 min / 58.9 s; MAST archive **>16 TB**, light curves **~3 TB** (https://archive.stsci.edu/missions-and-data/kepler ; https://phys.org/news/2018-11-mikulski-archive-space-telescopes-kepler.html). **Tens of TB — small.**
- **TESS (operating):** **>232,000 stars at 2-min** in the prime mission, **~1.5M 2-min light curves** processed, FFI cadence **30→10→200 s** across cycles + a 20-s fast mode; **~10M stars** photometered from FFIs; TIC-8 ≈ **1.5–1.7B point sources**. **~2 PB/year**; users pulled **680 TB in 2020 alone (~7× Hubble)** (https://archive.stsci.edu/missions-and-data/tess ; https://www.stsci.edu/contents/newsletters/2020-volume-37-issue-01/long-live-tess). **TESS is the largest *dense* time-series photometry archive today.**
- **CHEOPS (ESA):** single-target follow-up, not a survey; **~150 MB/day**; **extended to end-2029** (https://astrobiology.com/2026/06/16/cheops-exoplanet-mission-extended/ , 2026-07-14). Trivial volume.
- **PLATO (ESA):** **NOT yet launched** — built Oct 2025, launch **late 2026, likely slipping to Jan/Mar 2027**, Ariane 62 to L2 (https://www.dlr.de/en/latest/news/2025/plato-mission-to-launch-in-late-2026-onboard-ariane-6 ; https://en.wikipedia.org/wiki/PLATO_(spacecraft) , 2026-07-14). **26 cameras**, ~200,000 target stars, Earth-analog focus; **>100 Tbit/day raw on-board**, ~435 Gbit/day downlinked.
- **Roman (NASA):** **NOT yet launched** — launch **moved up to 2026-08-30** to L2 (https://spacepolicyonline.com/news/nasa-sets-launch-date-for-roman-space-telescope/ , 2026-07-14). Bulge microlensing → **>1,000 exoplanets**; **~4 PB/year, ~20 PB over 5 yr**.
- **Vera C. Rubin / LSST — the extreme case:** LSSTCam first light Apr 2025, alert stream Feb 2026, **the 10-yr LSST officially began 2026-06-30** (https://rubinobservatory.org/news/action-rubin-lsst-begins ; https://www.washington.edu/news/2026/06/30/rubin-observatory-legacy-survey-space-time-lsst/ , 2026-07-14). Official key-numbers (https://rubinobservatory.org/for-scientists/rubin-101/key-numbers , 2026-07-14): **~10 TB/night** (secondary sources say 15–20 TB raw); final catalog (DR11) **37B objects (20B galaxies + 17B stars)**, **7 trillion single-epoch sources**, **30 trillion forced measurements**, **2.1M visits**, **catalog DB ~15 PB**, **~60 PB raw / ~500 PB processed** over 10 yr. **Alerts: ~10M/night design; early-operations news cites ~7M/night** (https://www.techtimes.com/articles/319434/20260701/ , 2026-07-14) `[reconcilable as design-max vs early-ops]`. First data release **DR1 mid-2028**; Early Data Preview 2 due 2026-07-27.

**Where is the volume genuinely EXTREME vs merely large? — Two different axes:**
- **Raw volume / total detections / alert rate:** Rubin (**60 PB raw / 500 PB processed; 30 trillion measurements; ~7–10M alerts/night**) ≫ Roman (~20 PB/5 yr) ≫ TESS (~2 PB/yr) ≫ PLATO (downlink-limited) ≫ Kepler (tens of TB).
- **Dense, uniformly-sampled light curves (the axis that matters for finding transiting planets):** Rubin is **sparse — ~800 visits per object over 10 years.** The dense-cadence crown is **TESS today** (1.5M 2-min curves; FFI curves for ~10–20M stars), soon **PLATO** (2.5–25 s cadence). *Rubin is extreme in breadth; TESS/PLATO are extreme in depth.* **The queryable-light-curve-at-scale tooling gap sits precisely between the two.**

**Formats — FITS → cloud-native Parquet/Arrow migration is real and underway.** Rubin distributes **catalogs as Parquet**; **HATS** (Hierarchical Adaptive Tiling Scheme, formerly HiPSCat) stores each HEALPix sky tile as a Parquet file and **LSDB** (Large Survey DataBase, HATS + Dask) does distributed cross-match/analysis, with adopters including Rubin, STScI, IPAC, CDS, ESA (https://github.com/astronomy-commons/hats ; https://arxiv.org/abs/2501.02103 , 2026-07-14). The IVOA published the **VOParquet Note (2025-01-16)** and **astropy 8.0 added `parquet.votable`** (https://www.ivoa.net/documents/Notes/VOParquet/20250116/ , 2026-07-14).

**Is "cheap queryable photometry at scale" unsolved? Underserved, but *actively* being solved — this is the key honesty point.** The traditional **MAST/Lightkurve** model is per-target search-and-download and does *not* scale to bulk time-series analytics — which is why community FFI pipelines exist (QLP: "5.7 million stars," https://arxiv.org/abs/2211.04386 , 2026-07-14). But the LSDB team built **nested-pandas** for exactly this pain (joining sparse metadata to dense photometry) and computed a **periodogram for a billion ZTF light curves in 2 hours on 7 nodes** (https://arxiv.org/abs/2501.02103 , 2026-07-14). **Verdict for the engine:** the raw "massive time-series on cheap infra" pillar transfers technically, but **LSDB/HATS is a strong, funded, Rubin-adjacent incumbent in this exact lane.** In satellites the time-series competitors were mere propagators (Skyfield, SGP4); here there is a real cloud-native analytics platform. *Do not lead with "cheap queryable light curves" as whitespace — you would be racing LINCC Frameworks.* Lead with what rides *on top* of clean identities (§3–4).

## 3. The "where to find life" data methods — habitability + observability

Three metric families decide "where to look," and **every one of them is computed from parameters that come from the disagreeing catalogs of §1.** That is the crux.

**(a) Habitable Zone (HZ).** Kopparapu et al. 2013 (https://arxiv.org/abs/1301.6674 , 2026-07-14) sets effective-flux boundaries — Recent Venus (S_eff≈1.78), Runaway Greenhouse (1.05), Moist Greenhouse (1.01), Maximum Greenhouse (0.34), Early Mars (0.32) at solar T_eff — with a quartic T_eff dependence, and distance **d = √[(L/L⊙)/S_eff]**. **Conservative HZ** = runaway→maximum greenhouse; **optimistic HZ** = recent Venus→early Mars. **HZ is set entirely by catalog-derived stellar luminosity and T_eff.** The decisive result for the thesis is **Kane 2014, "Habitable Zone Dependence on Stellar Parameter Uncertainties"** (https://arxiv.org/abs/1401.3349 , 2026-07-14): a **~5% T_eff error → ~10% shift in HZ boundary**, and for 1,852 Kepler candidates the HZ uncertainty region is **200–400% of the HZ width** — i.e., **whether a planet is even *in* the habitable zone can flip depending on which catalog's stellar parameters you use.** This is the single strongest evidence that habitability rankings are provenance-sensitive.

**(b) Earth Similarity Index (ESI).** Schulze-Makuch et al. 2011: a geometric mean over radius, density, escape velocity, surface temperature vs Earth reference values (https://phl.upr.edu/projects/earth-similarity-index-esi , 2026-07-14) `[exact weight exponents unverified this pass]`. Widely used, widely criticized: **ESI is "not a direct measure of habitability"** — Venus and hot rocky planets score high — so it is a *filter*, not an answer.

**(c) Habitability Index (HITE).** Barnes et al. 2015 (https://arxiv.org/abs/1509.08922 , 2026-07-14): built around the eccentricity–albedo degeneracy; planets receiving **60–90% of Earth's insolation** are most likely habitable. Inputs are again catalog stellar radius, T_eff, log g.

**(d) Observability metrics — TSM/ESM — the operational decision layer.** Kempton et al. 2018 (https://arxiv.org/abs/1805.03671 , 2026-07-14):
- **TSM = (scale) × (R_p³ · T_eq) / (M_p · R⋆²) × 10^(−m_J/5)** — expected transmission-spectroscopy S/N.
- **ESM = 4.29×10⁶ × [B_7.5μm(T_day)/B_7.5μm(T⋆)] × (R_p/R⋆)² × 10^(−m_K/5)** — emission S/N at 7.5 μm.
- **Thresholds:** terrestrial (R_p<1.5 R⊕) **TSM>10**; larger **TSM>90**; **ESM benchmark 7.5.**
These are **the** operational currency for JWST atmospheric target selection: STScI's ~500-hr **Rocky Worlds DDT** (2024) explicitly bins targets in T_eq–R_p and ranks by TSM/ESM (https://www.stsci.edu/contents/news/jwst/2024/rocky-worlds-ddt-selects-its-first-targets , 2026-07-14), and Hord et al. 2023's "top TESS objects for JWST" list is built the same way (https://arxiv.org/abs/2308.09617 , 2026-07-14). **Every TSM/ESM input — R_p, M_p, T_eq, R⋆, J/K magnitudes — is pulled from the same disagreeing catalogs, and Hord et al. explicitly warn that "the most precise set is often chosen, [but] there is no guarantee it is most accurate, and the solutions may disagree."** Ranking is provenance-sensitive by construction.

**(e) The Habitable Worlds Observatory (HWO) target-selection problem — the contested, high-stakes core.**
- **ExEP Mission Star List / precursor stars: 164 "best" targets** for a ~6 m HWO.
- **HPIC (HWO Preliminary Input Catalog)**, Tuchow, Stark et al. 2024 (https://arxiv.org/abs/2402.08038 , 2026-07-14): **12,944 (~13,000) nearby bright stars**, auto-compiled by **cross-matching TIC 8.2 + Gaia DR3/DR2 + Bailer-Jones distances + SIMBAD + 2MASS + Washington Double Star + Gaia Catalog of Nearby Stars**, deriving T_eff, L, R, [Fe/H], mass, age, log g. **This is an entity-resolution + parameter-composite pipeline — the exact shape of the satellite engine.**
- **TSS25 (HWO Target Stars & Systems 2025)**, Tuchow, Stark et al. (https://arxiv.org/abs/2509.20544 , 2026-07-14): **12,944 stars in 3 tiers — Tier 1 = 164 (ExEP list), Tier 2 = 495, Tier 3 = 12,285**; 659 (Tiers 1+2) recommended for precursor characterization. The mission goal is **~25 exo-Earths** surveyed from the **~100 nearest systems** (Astro2020 decadal). Yield modeling predicts only **~200–400 stars** would actually be observed.
- **Yield robustness:** Stark et al. 2024 (https://arxiv.org/abs/2405.19418 , 2026-07-14) finds **η⊕ (planet occurrence) uncertainty dominates** exo-Earth yield uncertainty; **P(≥25 EECs) is only ~6%** without η⊕ margin (~32% with) — which is why the program buys margin via a *bigger telescope*, not parameter refinement.

**Is target selection genuinely contested over catalog-provenance parameters? — Yes at the star level, and this is the opening.** The *aggregate* "~25" is engineered to be **robust** to random stellar-parameter noise (yield varies only ~5%; 10–20% precision "sufficient… so long as there are no systematic population-level biases"). But **which individual stars rank into the prioritized ~25–164 is not agreed and is provenance-sensitive:** (i) TSS25's Tier 2 had to **union two independent yield codes (AYO vs EXOSIMS) because their per-star rankings disagree**; (ii) HZ membership — hence which stars host detectable HZ Earths — **moves directly with catalog R⋆/T_eff/distance and with the conservative-vs-optimistic HZ choice** (Kane 2014); (iii) the community deliberately ships a **tiered, living, updatable** list rather than one canonical answer — itself an admission that star-level prioritization depends on input provenance, yield method, and HZ definition. **Crucially: I found no paper that quantifies rank-order churn of the top ~25 as a direct function of catalog choice** — the evidence it *would* churn is strong but indirect (dual-code unioning + Kane 2014). `[This "could not find" is the core novelty claim — strong but rebuttable, exactly like landscape.md §3.1.]`

## 4. Honest verdict

### 4a. Is there a novel data/engineering angle nobody occupies? — Yes: *provenance-propagated target re-ranking.* Ranked candidates:

| Rank | Angle | Novelty | Solo-feasible | Who cares | What exists today | Why it's open |
|---|---|---|---|---|---|---|
| **1** | **Conflict-aware, provenance-tracked *re-ranking* of JWST TSM/ESM + HWO habitability targets — "does the rank flip depending on which catalog/publication you trust?"** Pull *every* published parameter from the PS table + EU + Gaia/TIC, propagate the disagreement through TSM/ESM/HZ/ESI, and report which targets cross a threshold (TSM>90, ESM>7.5, in/out of HZ) or change rank order under different sources. | **Very high** | **High** (top tiers are small: 164–659 stars, a few hundred JWST-relevant TOIs; data is public + license-clean) | HWO **START**/ExoPAG target teams; JWST proposers; NExScI | Exo-MerCat *flattens* to one value; Hord 2023/TSS25 publish **single** ranked lists; Kane 2014 shows membership flips — **but nobody exposes rank-order churn as a function of provenance** | Pure execution gap: the PS table already holds the disagreement; no tool turns it into a provenance-tracked, re-rankable conflict report feeding habitability+observability metrics |
| **2** | **A `merge_log` for exoplanet host stars** — per-attribute `source_assertion` across NASA PS rows + EU + OEC + Gaia DR3/TIC/HD/HIP/2MASS, with an auditable conflict log (the satellite engine's signature move, ported). | **High** | **High** | NExScI; catalog maintainers; anyone doing population studies | Exo-MerCat (SIMBAD main-ID + lowest-error) is adjacent but flattens; SIMBAD/CDS/ExoFOP resolve IDs, not per-attribute provenance | The *auditable, conflict-exposing, temporally-versioned* combination is unmet, same as satellites — but Exo-MerCat is a closer competitor than anything in the satellite lane |
| **3** | **MCP-native, provenance-aware exoplanet identity/target service** — expose #1/#2 as an MCP server so an LLM can ask "rank the HWO Tier-1 stars by TSM and tell me which flips if I use the discovery-paper radius instead of the Gaia radius." | **High** | **High** | Analysts, agents, hiring-manager demos, ExoPAG SIGs | No provenance-aware exoplanet MCP server found `[could-not-find]` | Same wedge that ranked #1 in the satellite landscape (`landscape.md` §3.3); the data is public, nobody has wrapped a *reconciled, provenance-aware* target ranker in the agent protocol |
| 4 | **Cheap queryable light-curve layer** (Parquet/HATS over TESS/Kepler FFIs). | **Low** | Medium | Time-domain researchers | **LSDB/HATS/nested-pandas already do this, funded** (§2) | Effectively closed — do not lead here |

**Top pick: #1.** It is the direct fusion of all three engine pillars (entity resolution across disagreeing catalogs → clean identities → metrics that ride on them → a conflict report) onto the field's single most consequential decision. It rhymes with the satellite project's strongest differentiator — *conflict reporting as a first-class product* (`landscape.md` §3.1.2) — and lands it on "which stars does a multi-billion-dollar telescope observe to find life." The novelty is a genuine, verified-by-absence execution gap; the feasibility is high because the decision-relevant N is *small* (hundreds, not billions).

### 4b. Career/commercial reality — **academic-and-scarce, NOT real money (in exoplanets). Be clear-eyed: the revenue stays on the satellite side you already serve.**

- **Employment is real but government/academic-salaried, not venture-funded.** Employers: **STScI/MAST, IPAC/NExScI (Caltech), NASA GSFC/JPL, NOIRLab**, university survey teams (Rubin). Astronomy software engineers average **~$129,678** (https://www.ziprecruiter.com/Jobs/Astronomy-Software-Engineer , 2026-07-14); STScI posts senior science-SWE at **$120–135k** (https://spacecrew.com/space-jobs/ , 2026-07-14) — at/near market, below FAANG. ~two-thirds of new astronomy PhDs go to the **private sector** (median start ~$120k) against a saturated academic market with ~4× more postdocs than faculty slots (https://www.aip.org/statistics/new-astronomy-phds-what-comes-next , 2026-07-14).
- **The commercial contrast is the whole point.** Satellite/SSA data *has* money: SSA market **~$1.7–1.9B** in 2026 (https://www.grandviewresearch.com/industry-analysis/space-situational-awareness-market-report , 2026-07-14), commercial EO **~$5.1B**. Exoplanet data has **essentially zero** commercial market: the lone "astronomy-as-a-service" play, UK's **Blue Skies Space**, has raised only **~$6.5M** (https://techcrunch.com/2024/08/07/uk-satellite-startup-blue-skies-space-wants-to-sell-astronomy-data-as-a-service/ , 2026-07-14). (Irony worth flagging: the profitable company named **"Astronomer"** — $39.5M ARR — is an Apache-Airflow data-pipeline firm with *nothing to do with astronomy*, https://getlatka.com/companies/astronomer , 2026-07-14. The money is in the data engineering, not the astronomy.)
- **Who would specifically care about angle #1:** the **HWO START team** (co-chairs Courtney Dressing / John O'Meara, 56 members, incl. a Living Worlds WG; https://science.nasa.gov/astrophysics/programs/habitable-worlds-observatory/start-tag/ , 2026-07-14); **ExoPAG** SIGs/SAGs; **NExScI/IPAC** (whose new Stellar Host Parameters table explicitly serves HWO/Roman vetting); named researchers **Chris Stark, Ravi Kopparapu, Noah Tuchow** (GSFC) and **Eliza Kempton** — Tuchow's CHZ2 habitable-zone metric for prioritizing HWO targets (https://arxiv.org/abs/2505.20558 , 2026-07-14) is almost a spec for this tool.
- **Funding vehicle is grants, not customers:** NASA ROSES **XRP** (Exoplanets Research) and **ADAP** (Astrophysics Data Analysis — a natural fit for a reconciliation engine); realistically a solo unaffiliated dev enters as **Co-I under an academic PI** `[solo PI eligibility unverified]`.
- **The one place these exact skills *are* operationally valued and hiring:** the **Rubin/LSST alert-broker ecosystem** — ALeRCE, AMPEL, ANTARES, Fink, Lasair, Pitt-Google — processing ~7M alerts/night. **ANTARES (NOIRLab) is architecturally an entity-resolution + time-series system:** it builds a per-position "locus," cross-matches against catalog "AstroObjects," and *duplicates* an alert when identity "cannot be uniquely determined" (https://iopscience.iop.org/article/10.3847/1538-3881/abd703 , 2026-07-14). That is the satellite engine's competency, in astronomy, with jobs attached.

**Honest ranking of solo-dev outcomes:** (a) **portfolio/credibility piece — highly plausible**; (b) **employment at STScI/IPAC/NOIRLab or a broker team — plausible, the best "money" outcome, below FAANG**; (c) **research co-authorship** with Stark/Tuchow/Kempton via a working tool — **plausible**; (d) **open-source adoption — possible but hard** (competes with the entrenched Archive + Exo-MerCat); (e) **grant funding — possible, structurally hard solo**; (f) **startup/revenue — fantasy for exoplanets.** Bottom line: **exoplanet work buys credibility, collaborators, and possibly a job — not revenue. The commercially rational play is to keep the exoplanet build as a credibility/hiring/collaboration asset while monetizing the same engine on the satellite/SSA data you already have.**

### 4c. Minimal, high-signal first module (~1 week) — *"Does the target rank flip?"*
